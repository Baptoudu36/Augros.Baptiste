Abstract. DEPOXY investigates how local muscle oxygenation (mNIRS, Train.red) relates to pulmonary oxygen uptake (VO2p, COSMED K5) during submaximal treadmill running under three footwear conditions in an ABCCBA crossover design. Four asynchronous modalities — metabolic (K5), muscle oxygenation (mNIRS over vastus lateralis and medial gastrocnemius), ground reaction forces (instrumented treadmill), and heart rate — are fused through a modular Python pipeline (load → sync → filter → segment → features) before statistical inference is performed jointly in Python (statistics.py) and R (main.Rmd). This report documents the full technical architecture: schema-normalised ingestion, trigger-based synchronisation with automated fallback, auto-tuned zero-phase filtering, protocol-driven phase segmentation, and feature extraction per block and per condition. The pipeline has been validated end-to-end on the pre-test participant dataset and is designed to scale to the ~20-participant target cohort with no structural change.

1. Project Context and Objectives

Pulmonary VO2 is the reference indicator of running economy but it aggregates the oxidative demand of the whole organism, masking muscle-specific contributions. Portable mNIRS devices now provide continuous, non-invasive access to local O2 delivery–extraction balance (TSI, O2Hb, HHb, tHb) at the muscle site. The DEPOXY project combines both modalities to test whether local oxygenation indicators derived from mNIRS can explain or predict variations in global energy cost when running shoes are changed — a question of direct interest to both the exercise physiology literature (Perrey et al., 2024) and the industrial partner (Decathlon R&D).

The present deliverable is a technical report. Its primary focus is the data-analysis pipeline: ingestion, synchronisation, filtering, segmentation, feature extraction and statistical infrastructure. Physiological interpretation is kept deliberately brief; cohort-level inference will be reported in the Master's dissertation once acquisition is complete (full cohort: ~20 participants).

2. Experimental Design

2.1 Protocol timeline

Each session lasts approximately 75 minutes and follows a strictly time-locked template encoded in segmentation.py. The six 5-minute running blocks follow an ABCCBA mirror-crossover order, so that each footwear condition is executed twice at symmetric positions in the protocol. This design balances fatigue, familiarisation and order effects across the three conditions.

Core technical challenge. The four devices run on independent clocks and at sampling rates that differ by up to four orders of magnitude (≈0.3 Hz vs. 1000 Hz). A dedicated synchronisation stage is therefore required before any cross-modal comparison is possible.

3.1 Data ingestion — load_data.py

Each CSV uses a different header layout, separator and decimal convention. Dedicated loaders (load_k5_data, load_nirs_data, load_force_plate_data, load_hr_data) auto-detect the header row (e.g. the K5 export hides the t column after multiple metadata rows; the Train.red NIRS file starts its tabular block at line 61), convert European-format decimals with _european_to_float(), and emit a canonical schema per modality (time_sec, VO2_ml_min, TSI_Quad, HHb_Quad, total_force_N, HR_bpm, …). Oxyhaemoglobin is derived as O2Hb = HbTot − HHb. Force-plate voltages are converted to Newtons using a calibration factor (FORCE_CALIBRATION_FACTOR = 100 N/V). The K5 signal is additionally reindexed on a 1-second grid by _impute_k5_time_grid(), with linear interpolation between breaths and padding up to the 75-minute protocol length.

3.2 Synchronisation — synchronization.py

The COSMED K5 event channel is the temporal master. find_trigger_time() locates the first non-zero event as t=0; every other modality is then shifted to the same origin. Because an operator can forget to press a trigger or the channel can be noisy, a robust two-level fallback is implemented:

PriorityStrategyTriggered when…
1Explicit event column on each modalityK5 / NIRS / FP have a usable event column
2infer_manual_trigger_time() — signal-drivenNo event present: detect first sustained deviation in the derivative and level of a modality-specific channel (MAD-robust thresholds)
3K5 offset reuseInference also fails: use the K5 t0 as a wall-clock proxy

Table 3. Cascading trigger-detection strategy implemented in synchronize_all().

Pre-trigger samples are dropped (drop_pre_trigger=True). A helper resample_to_common_grid() is available when correlation analyses require a single aligned grid; the default pipeline keeps each modality on its native time axis to preserve information.

3.3 Filtering — filtering.py

All filtering is performed with zero-phase Butterworth filters (scipy.signal.butter + filtfilt) to avoid phase distortion of physiological trends. Cut-off frequencies are not hard-coded: a composite quality score (noise reduction minus amplitude-collapse penalty) is computed over a representative channel for each modality and the best candidate is selected automatically.

3.4 Phase segmentation — segmentation.py

The protocol timeline is expressed programmatically by create_protocol_timeline(): a dictionary of phase windows derived from the PROTOCOL constants (baseline 300 s, warmup 300 s, pre-block recovery 300 s, 6 runs of 300 s with 300 s recoveries between them, final baseline 300 s). segment_all_phases() labels every sample with its phase identifier. A dedicated extract_steady_state_from_block() carves out the 2:00–4:30 window of each run — discarding the first 2 minutes (metabolic adaptation) and the final 30 seconds (anticipatory withdrawal) — which is the physiologically defensible window for steady-state averaging.

# Steady-state extraction (segmentation.py)
def extract_steady_state_from_block(df, block_start_sec,
                                    ss_start_offset=120,
                                    ss_end_offset=270):
    ss_start = block_start_sec + ss_start_offset   # 2:00
    ss_end   = block_start_sec + ss_end_offset     # 4:30
    return extract_window(df, ss_start, ss_end), {...}

4. Statistical Analysis

Inference is split between Python and R along tool-strength lines: Python is used for descriptive tests, omnibus ANOVA, non-parametric backup and correlation matrices; R is used for linear mixed-effects models that accommodate the repeated-measures structure and inter-individual variability typical of running-economy research.

4.2 R side — main.Rmd

The R Markdown document fits linear mixed-effects models with lme4/lmerTest (fixed: footwear condition; random: participant intercept), complemented by afex::aov_car for classical repeated-measures ANOVA and emmeans for Bonferroni-corrected pairwise contrasts. Outputs are rendered to a self-contained HTML report via rmarkdown::render(). LMM are the appropriate framework here because they accommodate (i) the within-participant structure (two exposures per shoe), (ii) missing segments from occasional signal dropouts without listwise deletion, and (iii) the substantial inter-individual baseline variability expected in running economy.

6. Reproducibility and Pipeline Validation

The project is reproducible from a clean clone of the repository. A conda environment (environment.yml) and a pip fallback (requirements.txt) pin the scientific stack (numpy 1.26, pandas 2.2, scipy 1.13, scikit-learn 1.4, statsmodels 0.14, pingouin 0.5, matplotlib 3.8, seaborn 0.13). Each Python module exposes a __main__ validation block that loads the pre-test data, runs its stage, and asserts structural invariants (row counts, timeline coverage, condition order, column presence).

# Reproduce
git clone https://github.com/Baptoudu36/Augros.Baptiste.git
cd Augros.Baptiste
conda env create -f environment.yml && conda activate depoxy
jupyter notebook main.ipynb               # Python pipeline + figures
Rscript -e "rmarkdown::render('main.Rmd')" # R inferential report

Validation on the pre-test participant dataset confirms that the four modalities ingest cleanly, synchronise to a common origin, filter without edge artefacts, segment into the expected ten non-empty phases (2 baselines + warmup + pre-block recovery + 6 runs + 5 inter-block recoveries), and produce the expected 6-row block table and 3-row condition table (A, B, C). Missing triggers in the heart-rate file were correctly recovered by the signal-driven inference fallback.

Data handling note. Participant data are pseudonymised by a unique code (000NnPp format) in line with the laboratory's Data Management Plan (EuroMov DHM / UM). The raw force-plate file exceeds GitHub's size cap and is hosted separately (Google Drive link in README.md). 7. Limitations and Perspectives

Three principal limitations should be flagged for the defence. First, the pipeline has been validated end-to-end only on the pre-test participant; cohort-level acquisition was completed on 16 April 2026 and integration of the full ~20-participant dataset is ongoing. Second, force-plate calibration relies on a nominal 100 N/V factor; a per-session calibration sheet would tighten the conversion but does not affect relative condition contrasts, which are the primary analytical target. Third, no electromyographic channel is recorded: the interpretation of coupled changes in HHb and step frequency remains indirect.

Two extensions are planned. (i) A signal-quality dashboard that flags dropped NIRS frames, K5 breath outliers and force-plate saturation per block, enabling per-block quality weighting in the LMM. (ii) Integration of the VO2/HHb ratio as a candidate composite indicator of peripheral oxygen cost, to be tested against condition contrasts in the final dissertation.
